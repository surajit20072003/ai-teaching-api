# Data Architecture & Pre-Generation Deep Dive

This document provides a detailed breakdown of how the AI Teaching API handles data, caching, database interactions, and the asynchronous pre-generation pipeline.

---

## 1. Databases Used

The system relies on a dual-database architecture:

### A. PostgreSQL (The Primary Datastore)
- **Role:** Persistent storage for all curriculum structures (subjects, chapters, topics), raw questions, RAG documents, and the finalized AI output cache.
- **Extensions:** Uses `pgvector` for storing high-dimensional embeddings (384 dimensions) generated from document text chunks.
- **Driver:** Uses `asyncpg` combined with `SQLAlchemy` for fully non-blocking, asynchronous database queries.
- **Capacity/Scale:** 
  - Standard PostgreSQL handles hundreds of millions of rows easily. 
  - The `teaching_qa_cache` stores the large JSON payloads (slide text, audio URLs). Thanks to `JSONB` binary storage, queries against these nested structures remain extremely fast.
  - The `document_chunks` table handles vector similarity search. The `hnsw` (Hierarchical Navigable Small World) index is applied here, allowing sub-10ms similarity matching even across millions of textbook chunks.

### B. Redis (State & Concurrency Management)
- **Role:** Fast, in-memory key-value store used exclusively for managing the state of the active Pre-Generation batches.
- **Scale:** Redis can process >100,000 operations per second.
- **Usage:**
  - **Job Locks:** Prevents two identical pre-generation jobs from running simultaneously.
  - **State Tracking:** Tracks the `current_count`, `total_count`, and the active `phase` (Text vs. Media vs. Manim) so the frontend dashboard can poll the `/pregen/status` endpoint and show a real-time progress bar without querying PostgreSQL.

## 2. The Database Schema & Flow

### Curriculum & Raw Questions
1. **`subjects`, `chapters`, `topics`**: Defines the hierarchy. E.g., Math -> Algebra -> Quadratic Equations.
2. **`questions`**: The raw question bank. This stores the question text, difficulty, and marks. 
   - **Crucial Flag:** `is_pregen_done` defaults to `FALSE`. 
   - Once a question successfully passes through the pre-generation pipeline, `is_pregen_done` is flipped to `TRUE`, and the `cache_id` is populated.

### The Cache: `teaching_qa_cache`
This is the most critical table for user-facing speed. When a student's app requests a question, it queries *this* table, not the raw `questions` table.

**Table Structure:**
- `id` (UUID): Primary key.
- `question_hash` (TEXT): An MD5/SHA hash of the question text. This ensures we never duplicate generation for the exact same question, even if it appears in multiple chapters.
- `presentation_slides` (JSONB): The massive JSON array generated by Ollama. Contains every slide's text, equations, and script.
- `slide_audio_urls` (JSONB) / `video_url`: URLs pointing to the final MP4s or WAVs on the local disk or Backblaze B2.
- `usage_count` (INT): Tracks how many times students have viewed this specific cache entry (useful for analytics).

**How the Cache Works:**
1. A student app asks for a question.
2. The API checks if `teaching_qa_cache` has an entry for this question.
3. If **YES**: It returns the JSON payload in ~20 milliseconds.
4. If **NO**: The API *could* generate it live (which takes ~3 minutes), but the entire purpose of this project is to use the **Pregen Pipeline** to ensure the cache is always filled *before* the student ever asks.

## 3. The Detailed Pre-Generation Pipeline

The pre-generation pipeline (`core/pregen.py`) runs as a background task. It scans the `questions` table for any row where `is_pregen_done = FALSE` and processes them in three strict, atomic phases to protect GPU VRAM.

### Data Flow during Pregen

#### Step 1: Initialization
- A batch is started via the API.
- Redis creates a state lock: `{"running": true, "phase": "Initializing", "total": X}`.
- The system grabs up to `limit` (e.g., 500) questions from the database.

#### Phase A: Text & JSON Generation (GPU: Ollama)
- For each question, the system queries the `document_chunks` table using `pgvector` to find relevant textbook paragraphs.
- It sends the question + textbook context to the local Ollama LLM.
- Ollama returns a JSON array of slides.
- **Database Action:** An incomplete record is inserted into `teaching_qa_cache` containing the slide text, but media URLs are empty.

#### Phase B: Media Generation (GPU: Wan2GP)
- **VRAM Eviction:** The system sends an API call to Ollama to unload the LLM from GPU memory. This is critical because Wan2GP (Image Gen) requires massive VRAM.
- The system reads the incomplete `teaching_qa_cache` rows.
- It sends background requests to VoxCPM (audio) and Wan2GP (images).
- **Storage Action:** The returned `.wav` and `.png` files are saved to the local NVMe SSD (e.g., `/sdb-disk/ai-teaching/...`).
- **Database Action:** The `teaching_qa_cache` is updated with the local file paths. An async background task uploads these files to Backblaze B2 and updates the database with the public B2 URLs.

#### Phase C: Manim Animations (CPU / API)
- The system scans the cache rows for slides marked as `visual_type: "manim"`.
- It connects to OpenRouter (`deepseek-chat`) to write a Python Manim script.
- The Python script is executed locally inside a subprocess to render an `.mp4`.
- **Storage Action:** The `.mp4` is saved to local NVMe, uploaded to B2, and the `presentation_slides` JSON block in `teaching_qa_cache` is updated with the video URL.

#### Finalization
- The `questions` table is updated: `is_pregen_done = TRUE` and `cache_id` is set to the newly completed row in `teaching_qa_cache`.
- Redis state is cleared.

## 4. Storage Capacities & File Management

### Local Disk (`/sdb-disk/ai-teaching/`)
- Acts as the scratchpad and initial cache.
- NVMe drives handle the extremely high IOPS required by Manim video rendering and Wan2GP image generation.
- **Structure:** `subjects/{subject_id}/cache/jobs/{job_id}/images/` and `.../audio/`.

### Cloud Storage (Backblaze B2)
- Serves as the CDN / permanent storage for all student apps.
- **Scale:** Practically infinite scale. 
- Files are named securely using UUIDs. Once the local system uploads a file to B2, the local server serves the B2 URL to the frontend, offloading bandwidth from the API server.

### Automatic Retry (Layer 2)
Because external media APIs drop connections, the DB might have rows where Text exists but audio failed.
- The `/pregen/retry-media` endpoint queries `teaching_qa_cache` directly.
- It parses the `presentation_slides` JSON.
- If a slide requires an image, but the `image_url` is missing, it skips Phase A entirely and jumps straight to Phase B to request *only* that missing image. This saves thousands of GPU hours at scale.
